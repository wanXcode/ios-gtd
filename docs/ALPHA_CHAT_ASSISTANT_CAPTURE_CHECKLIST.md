# Alpha 联调清单：当前飞书对话里怎么试 assistant / capture

这份清单只服务当前 alpha 联调，目标不是覆盖全部产品能力，而是让用户在**当前飞书对话框**里能稳定试出：

- 哪些话可以直接建
- 哪些话当前应该先追问
- 当前实现和目标合同之间还有哪些明确缺口

---

## 1. 先记住当前阶段的硬限制

当前后端真实合同以代码为准，不以旧文档为准：

- 接口是 `POST /api/assistant/capture`
- 请求字段是 `input`
- 控制是否落库的字段是 `apply`，不是 `dry_run`
- 当前返回只有 `draft / applied / created`
- 当前没有 `questions`
- 当前没有标准化 `error_code`
- 当前如果你直接 `apply=true`，即便 `draft.needs_confirmation=true`，后端也还会继续写库

所以 **飞书对话层当前正确联调方式** 是：

1. 先把用户原话发给 assistant 做解析（等价于后端 `apply=false`）
2. 如果结果明确，再由代理决定是否二次执行创建（等价于后端 `apply=true`）
3. 如果结果显示 `needs_confirmation=true`，当前不要直接落库，应该先回问用户

一句话：**当前 alpha 先把 assistant 当“解析器 + 保守执行器”，不要当全自动秘书。**

---

## 2. Apple 列表基线（已确认，必须按精确字符串理解）

当前用户 Apple Reminders 列表精确字符串：

- `收集箱 @Inbox`
- `下一步行动@NextAction`
- `项目 @Project`
- `等待@Waiting For`
- `可能的事 @Maybe`

默认规则：

- **不确定时，一律先放 `收集箱 @Inbox`**

但也要知道：

- 当前 backend bucket 推断还比较保守
- alpha 阶段绝大多数 capture 实际仍更容易落到 backend 的 `inbox`
- 所以这轮联调更重要的是验证“是否该先收进 Inbox”，不是验证“是否已经能稳定自动分到 NextAction”

---

## 3. 在当前飞书对话里，建议这样试

### 3.1 先测“应该能直接记下”的话

你可以直接在当前飞书对话里发这些句子：

1. `给张三发合同`
2. `明晚8点给妈妈打电话`
3. `找时间整理一下财务系统`
4. `以后再说，研究一下 OCR`

这四句分别主要看：

- 是否能抽出稳定标题
- 是否不会出现空标题 / Untitled task
- 是否能对相对时间给出稳定解析
- 是否能把“以后再说”这种表达打到更偏 someday / maybe 的语义

**当前可接受结果：**

- 1、2 大概率能解析成功
- 3 更适合先进 Inbox
- 4 当前如果落成 someday / maybe 语义算符合预期；如果仍偏保守，也至少不该乱补具体时间

### 3.2 再测“应该先追问，不该直接建”的话

继续发：

1. `准备 Q2 产品升级方案`
2. `晚点提醒我看下邮箱`
3. `下周处理合同`
4. `明天给张三发合同，再给李四回电话`
5. `这个也记一下`

这几句分别主要看：

- 项目 / 任务是否有歧义
- 时间是否太模糊
- 是否包含多件事
- 是否标题不足

**当前 alpha 的正确处理原则：**

- assistant 可以先给出解析草稿
- 但主代理不应该直接执行创建
- 应该在飞书里继续追问用户一句

注意：当前后端本身还不会返回结构化 `questions`，所以这层追问文案要由飞书代理自己补

## 4. 当前飞书代理建议采用的最小对话策略

### A. 直接创建型

如果用户说：

- `给张三发合同`
- `明晚8点给妈妈打电话`

代理可以按下面思路处理：

1. 先 parse
2. 如果 `needs_confirmation=false`
3. 再 apply 创建
4. 在飞书里回复一句：
   - `已帮你记下：给张三发合同。`
   - `已帮你记下：明晚 8 点给妈妈打电话。`

### B. 先收 Inbox 型

如果用户说：

- `找时间整理一下财务系统`
- `先记一下优化 onboarding 文案`

代理建议回复：

- `已先记到 Inbox：整理一下财务系统。暂时没给它设具体时间。`
- `已先记到 Inbox：优化 onboarding 文案。`

### C. 必须追问型

如果用户说：

- `准备 Q2 产品升级方案`

代理当前建议追问：

- `这件事更像一个项目。你要我把它建成项目，还是先记成一条普通任务？`

如果用户说：

- `晚点提醒我看下邮箱`

代理当前建议追问：

- `可以记这件事，但“晚点”还不够具体。你想让我几点提醒你？`

如果用户说：

- `明天给张三发合同，再给李四回电话`

代理当前建议追问：

- `我检测到你这句话里有两件事。要我拆成两条任务吗？`

---

## 5. 这轮 alpha 联调，重点不要看错地方

这轮不是在验收“最终智能程度”，而是在验收下面 5 件事：

1. **接口合同有没有对齐**
   - `input`
   - `apply`
   - `draft / applied / created`

2. **飞书代理会不会误执行**
   - 尤其是 `needs_confirmation=true` 时不能直接 apply

3. **Apple 默认 Inbox 规则有没有贯彻**
   - 不确定先去 `收集箱 @Inbox`

4. **精确字符串有没有被文档和实现同时记住**
   - 列表名不能擅自换写法

5. **当前限制有没有被明确暴露出来**
   - 没有 questions
   - 没有标准错误码
   - bucket 还不够智能
   - 项目识别还很窄

---

## 6. 一份最小可执行 smoke 顺序

如果老板只想在当前飞书对话里快速过一遍，建议按这个顺序发：

1. `给张三发合同`
2. `明晚8点给妈妈打电话`
3. `找时间整理一下财务系统`
4. `准备 Q2 产品升级方案`
5. `晚点提醒我看下邮箱`
6. `明天给张三发合同，再给李四回电话`

预期：

- 前 3 条里，至少 1、2 应能稳定变成可创建草稿
- 第 3 条应更偏 Inbox 语义
- 后 3 条应进入“先问一句再说”的路径，而不是直接帮用户建错

---

## 7. 当前阶段建议是否可以继续往主线合

可以，但前提是把它当作：

- **alpha 对话层合同澄清**
- 不是“assistant 已完成产品化”的信号

更准确地说，当前这轮文档合入主线的价值在于：

- 把旧的 `dry_run` 说法纠正成现在真实的 `apply`
- 把“默认 Inbox”规则和 Apple 精确列表名固定下来
- 把“当前后端不会替你拦 `needs_confirmation=true`”这个坑明确写给联调方

如果这三点不进主线，后面联调会继续踩同一个坑。
