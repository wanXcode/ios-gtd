# AI 自然语言任务入口交互约定与验收清单（v0.2.0）

本文是对 `docs/AI_TASK_ENTRY_PLAN.md` 的开发落地补充，目标不是再讲“为什么做”，而是明确：

- v0.2.0 到底允许哪些自然语言直接创建
- 哪些场景必须追问，不允许静默落库
- 接口层/对话层应该返回什么模板
- 字段确认流程怎么走
- 联调与验收时要测哪些 case

本文默认面向三个读者：

1. backend 开发：确定 `/assistant/capture` 的行为边界
2. AI/Agent 接入层：确定什么情况下直接调用、什么情况下先追问
3. 测试：拿本文直接写用例，不靠口头理解

---

## 1. 文档定位与版本边界

### 1.1 v0.2.0 的目标

v0.2.0 只解决一件事：

> 自然语言输入 → 结构化任务草稿/确认问题 → 创建到 backend 主库

它不是完整的任务管理 AI，也不是完整的多轮计划系统。

### 1.2 v0.2.0 不解决的问题

以下能力不作为 v0.2.0 发布阻塞项：

- 自动拆解项目为多个子任务
- 自动推断复杂上下文（如“还是上次那个合同”）
- 批量整理 Inbox
- 多轮记忆式会话规划
- backend → Apple Reminders 产品化反向闭环

### 1.3 与当前实现的关系

当前仓库内已经有最小版：

- `POST /api/assistant/capture`
- `GET /api/assistant/views/today`
- `GET /api/assistant/views/waiting`

但当前 `capture` 仍是启发式解析，能力偏保守。本文定义的是 **v0.2.0 应收敛到的交互合同**。

也就是说：

- 当前实现未覆盖的，本文作为开发目标
- 当前实现已存在但行为不稳定的，本文作为统一标准

## 2. 设计原则

## 2.1 宁可少建，不要错建

对用户来说，“AI 帮我多问一句”通常比“AI 自作主张建错一条任务”可接受。

因此 v0.2.0 默认策略是：

- **标题明确、类型明确、时间明确** → 可直接创建
- **标题明确，但类型/时间不明确** → 允许先落 Inbox
- **任务/项目边界不明确，且误建代价较高** → 先追问
- **字段冲突或明显解析异常** → 不创建，返回错误/澄清

## 2.2 追问必须有目的

不允许为了“显得聪明”而追问。

只有当追问能显著降低误建率时才追问，主要集中在：

- 任务 vs 项目
- 是否需要明确时间
- 时间是截止还是提醒
- 归类 bucket 是否影响后续使用

## 2.3 v0.2.0 只做“单条输入单条结果”为主

默认每次输入只处理一条主要事项。像“明天给张三发合同，顺便约李四吃饭，再把报销单理一下”这种多任务句子，不作为 v0.2.0 必须一次性拆分的能力。

策略：

- 能稳定识别第一主任务时，可提示“检测到可能包含多件事，当前只创建第一件”
- 更推荐直接返回需确认，让用户拆开说

## 3. 输入分类：哪些直接创建，哪些要追问

## 3.1 可直接创建的输入

满足以下条件时，AI/后端可直接创建，不需要追问：

### A. 明确的单动作任务

特征：

- 有清晰动作动词
- 有明确执行对象
- 语义上像单个下一步行动

示例：

- 给张三发合同
- 回王总电话
- 提交报销单
- 整理会议纪要
- 预约牙医

默认落库：

- `intent = create_task`
- `bucket = next` 或 `inbox`（见 4.2）

### B. 明确任务 + 明确时间

示例：

- 明晚 8 点提醒我给张三发合同
- 明天下午给客户回电话
- 下周一前提交报销
- 今天 6 点前把日报发掉

默认行为：

- 直接创建 task
- 尽量提取 `due_at`
- 若句子显式出现“提醒我”，同时写入 `remind_at`

### C. 模糊时间但适合进入 Inbox 的捕获型输入

示例：

- 找时间整理一下财务系统
- 之后研究一下 OCR
- 有空看看合同模板
- 先记一下：优化 onboarding 文案

默认行为：

- 直接创建 task
- 不强行设置 `due_at`
- `bucket = inbox` 或 `someday`（见 4.2）
- 不因为“找时间”“之后”“有空”自动虚构具体时间

### D. 用户显式要求“记一下/帮我建个任务”

示例：

- 帮我记一个任务：测试一，明晚 8 点
- 建个待办：给供应商发邮件
- 帮我加个任务，下周处理合同

默认行为：

- 优先按 task 处理
- 除非内容明显是项目，否则不追问“是不是任务”

---

## 3.2 必须追问后再创建的输入

满足以下情况时，系统应返回 `needs_confirmation=true`，而不是直接落库。

### A. 任务 vs 项目边界明显不清

示例：

- 准备 Q2 产品升级方案
- 推进招聘流程优化
- 重构财务同步逻辑
- 搭建新的客户跟进体系

这类表达更像：

- 项目目标
- 一组工作的集合
- 需要拆解的事项

处理规则：

- 默认 **先追问**：这是要建成“项目”还是先记成“一条任务”？
- 不要在 v0.2.0 里静默建成普通 task

### B. 时间表达存在关键歧义

示例：

- 下周处理一下合同
- 晚点提醒我看下邮箱
- 尽快把方案发掉
- 周末弄一下报销

问题在于：

- “下周”具体到哪一天？
- “晚点”是几小时后还是今晚？
- “尽快”不是可执行时间字段
- “周末”是否要精确到某天某时？

处理规则：

- 如果不设时间也能作为 Inbox 捕获，则可直接建 Inbox
- 但如果用户用了“提醒我”，则提醒时间不能模糊，必须追问

例如：

- “下周处理一下合同” → 可直接入 Inbox，且提示“暂未设置具体时间”
- “晚点提醒我看下邮箱” → 必须追问“你想几点提醒？”

### C. due 与 remind 语义冲突

示例：

- 明天提醒我周五发合同
- 下周一截止，明晚提醒我处理报销
- 今天记得下周交材料

如果一句话中同时出现多个时间点，且无法稳定判断：

- 哪个是 `due_at`
- 哪个是 `remind_at`

则必须追问，而不是猜。

### D. 缺少可用标题

示例：

- 这个也记一下
- 别忘了那个事
- 还是上次那个

如果脱离上下文后无法形成一个明确标题，不允许直接创建 `Untitled task` 作为正式产品行为。

注意：

- 当前代码的兜底 `Untitled task` 只可视为开发态保底
- v0.2.0 正式交互不应把这当默认用户体验

### E. 单句中疑似包含多件独立事务

示例：

- 明天给张三发合同，再给李四回电话，顺便把报销单提交了
- 帮我记一下招聘、财务和 SEO 这三块都要跟一下

处理规则：

- 默认追问：“我检测到你提了多件事，要我拆成多条任务吗？”
- v0.2.0 不要求自动多任务拆分成功率

---

## 3.3 不应创建，直接返回错误/澄清的输入

### A. 输入为空或仅噪音

示例：

- 空字符串
- “嗯”
- “啊？”

返回：

- `error_code = invalid_input`
- 明确提示需要说出要记录的事项

### B. 时间解析明显非法

示例：

- 2 月 30 号提醒我交材料
- 昨天 25 点给我提醒

返回：

- `error_code = invalid_datetime`
- 不创建任务

### C. 字段组合违反约束

示例：

- 解析出 `remind_at > due_at` 且句义不能自洽
- 解析出已完成状态但没有对应的 task update 场景

返回：

- `error_code = invalid_task_draft`
- 不创建任务

## 4. 结构化字段约定

## 4.1 v0.2.0 最小结构化字段

v0.2.0 至少需要稳定产出以下字段：

```json
{
  "intent": "create_task",
  "summary": "给张三发合同",
  "description": null,
  "due_at": "2026-03-21T20:00:00+08:00",
  "remind_at": "2026-03-21T20:00:00+08:00",
  "priority": null,
  "bucket": "next",
  "confidence": 0.94,
  "needs_confirmation": false,
  "questions": []
}
```

字段说明：

- `intent`：v0.2.0 主要是 `create_task` / `create_project` / `capture_inbox`
- `summary`：最终标题，必填
- `description`：可空，v0.2.0 不强求从输入中抽长描述
- `due_at`：截止时间，可空
- `remind_at`：提醒时间，可空
- `priority`：可空
- `bucket`：`inbox | next | waiting | someday | project`
- `confidence`：0~1 浮点数
- `needs_confirmation`：是否必须追问
- `questions`：待追问问题列表

## 4.2 bucket 决策规则

v0.2.0 不追求完美 GTD 推断，但要统一最低限度规则。

### 直接判为 `next`

满足任一条件：

- 明确动作 + 明确对象
- 明确动作 + 明确时间
- 用户显式说“提醒我/明天做/今晚处理”之类短期执行表达

示例：

- 给张三发合同
- 明天下午回客户电话
- 今晚把周报发掉

### 判为 `inbox`

满足任一条件：

- 只有方向，没有明确执行时点
- 用户只是先收集，不要求立刻排程
- 时间语义模糊但可安全记录

示例：

- 之后研究一下 OCR
- 找时间整理一下财务系统
- 先记一下优化 onboarding 文案

### 判为 `waiting`

只在句子明确表达“等别人/等回复/等对方”时使用。

示例：

- 等张三回复合同
- 等法务确认条款

注意：

- “联系张三确认合同” 是动作，应该是 `next`
- “等张三回复合同” 才是 `waiting`

### 判为 `someday`

只在明显是“将来也许/不着急/先放着”时使用。

示例：

- 有空研究一下 OCR
- 以后再看看 Notion 迁移
- 这个不急，先记着

### 判为 `project`

只在用户明确说“建项目”或语义强烈指向项目，且已确认时使用。

示例：

- 帮我建个项目：Q2 产品升级
- 这是个项目，不是单个任务

---

## 4.3 时间字段规则

### 规则 1：不要虚构时间

如果用户没给明确时间，不要为了“结构完整”擅自补：

- 今天 18:00
- 本周五 18:00
- 明天上午 9:00

这种自动脑补在 v0.2.0 一律禁止。

### 规则 2：`提醒我` 优先产生 `remind_at`

示例：

- 明晚 8 点提醒我给张三发合同

建议解析：

- `summary = 给张三发合同`
- `remind_at = 明晚 20:00`
- `due_at = 明晚 20:00`（v0.2.0 可暂时允许同值）

但如果后续实现要区分：

- “提醒时间” ≠ “截止时间”

则应在 API 字段层允许二者独立。

### 规则 3：模糊时间可用于 Inbox，但不可伪装成精确提醒

例如：

- “下周处理一下合同”

可以：

- 创建 Inbox
- `due_at = null`
- 附带说明“未设置具体时间”

不可以：

- 自动设为“下周最后一天 18:00”且当成确定 due

### 规则 4：全天语义与精确时刻要可区分

例如：

- “明天交材料” 更像全天截止
- “明天晚上 8 点交材料” 才是精确时刻

建议后端保留：

- `due_at`
- `is_all_day_due`

v0.2.0 验收中应覆盖这一点。

## 5. 追问协议

## 5.1 何时返回 `needs_confirmation=true`

以下任一场景触发：

- 任务/项目不明确
- 时间需要澄清且会影响创建字段
- 句子包含多件事
- 标题抽取失败
- due/remind 冲突

建议响应结构：

```json
{
  "draft": {
    "intent": "create_project",
    "summary": "准备 Q2 产品升级方案",
    "bucket": "project",
    "due_at": null,
    "remind_at": null,
    "confidence": 0.58
  },
  "needs_confirmation": true,
  "questions": [
    {
      "field": "intent",
      "question": "这是要建成一个项目，还是先记成一条普通任务？",
      "options": ["project", "task"]
    }
  ]
}
```

## 5.2 追问数量限制

v0.2.0 单轮最多追问 2 个问题。

原因：

- 一次追 4~5 个问题会让用户放弃
- 先把最关键的不确定字段确认下来更重要

优先级建议：

1. `intent`（task / project）
2. `remind_at` / `due_at`
3. `bucket`
4. `priority`

## 5.3 追问措辞要求

追问必须：

- 简短
- 可直接回答
- 尽量给选项

好例子：

- 这是普通任务还是项目？
- 你想设具体提醒时间吗？
- 这条先放 Inbox，还是直接作为 Next Action？

坏例子：

- 你是否希望我从 GTD 方法论角度进一步确认这件事的行动属性？

## 6. 回复模板

这里的“模板”主要给 AI/对话层用，backend 可以不直接返回自然语言，但需要返回能支撑这些模板的结构化字段。

## 6.1 直接创建成功模板

### 模板 A：有明确时间

> 已帮你记下：{summary}。时间是 {time_text}。

示例：

> 已帮你记下：给张三发合同。时间是明晚 8:00。

### 模板 B：无明确时间，进入 Inbox

> 已先记到 Inbox：{summary}。暂时没给它设具体时间。

示例：

> 已先记到 Inbox：整理财务系统。暂时没给它设具体时间。

### 模板 C：Waiting

> 已记下：{summary}，我先把它放到 Waiting。

---

## 6.2 追问模板

### 模板 A：任务 / 项目确认

> 这件事更像一个项目。你要我把它建成项目，还是先记成一条普通任务？

### 模板 B：提醒时间确认

> 可以记这件事，但“{time_expression}”还不够具体。你想让我几点提醒你？

### 模板 C：多任务拆分确认

> 我检测到这句话里可能有多件事。要我拆成多条任务吗？

### 模板 D：bucket 确认

> 这条我可以先记下来。你希望它先放 Inbox，还是直接作为 Next Action？

---

## 6.3 错误模板

### 模板 A：输入无效

> 我还没识别出你要记录的事项。你可以直接说一句，比如“明晚 8 点提醒我给张三发合同”。

### 模板 B：时间非法

> 这个时间我没法确认，像是无效日期。你可以换个更明确的时间再说一次。

### 模板 C：创建失败

> 我理解出任务了，但写入失败了。你可以稍后重试一次。

### 模板 D：解析异常

> 这句话里我没法稳定判断时间和任务类型，先不替你乱建。你可以换一种更直接的说法。

---

## 7. 字段确认流程

## 7.1 单轮创建流程

### 路径 A：直接创建

1. 用户输入自然语言
2. parser 产出 draft
3. policy 判断 `needs_confirmation=false`
4. 调用写入层创建 task/project
5. 返回创建结果 + 人类可读确认文案

### 路径 B：需要追问

1. 用户输入自然语言
2. parser 产出 draft
3. policy 判断 `needs_confirmation=true`
4. 返回问题列表，不落库
5. 等用户补充字段后再调用 apply/create

---

## 7.2 推荐的字段确认顺序

如果进入追问流程，字段确认顺序建议固定：

1. `intent`
2. `summary`
3. `due_at / remind_at`
4. `bucket`
5. `priority`

原因：

- `intent` 错了，后面字段都可能白问
- `summary` 不成立，整个任务对象都不成立
- 时间影响是否需要提醒、今日视图、Apple 映射

---

## 7.3 apply 前的最小校验

真正写库前，必须保证：

- `summary` 非空
- `intent` 明确
- 若有 `remind_at`，则时间合法
- 若 `bucket=project`，则应写 project 流程，不要仍走普通 task create
- 若 `needs_confirmation=true`，禁止 apply

建议写层前的最终 payload 至少满足：

```json
{
  "summary": "给张三发合同",
  "description": "",
  "due_at": "2026-03-21T20:00:00+08:00",
  "remind_at": "2026-03-21T20:00:00+08:00",
  "bucket": "next",
  "priority": null,
  "source": "chat_ai",
  "source_ref": "om_xxx"
}
```

## 8. API/后端返回合同建议

为了支撑上面的交互，建议把 `/assistant/capture` 从当前最小返回扩展为两段式语义：

### 8.1 dry-run 模式：只解析不写库

请求：

```json
{
  "input": "明晚8点提醒我给张三发合同",
  "context": {
    "timezone": "Asia/Shanghai",
    "source": "chat_ai",
    "source_ref": "om_xxx",
    "actor": "assistant"
  },
  "dry_run": true
}
```

响应建议：

```json
{
  "draft": {
    "intent": "create_task",
    "summary": "给张三发合同",
    "description": null,
    "due_at": "2026-03-21T20:00:00+08:00",
    "remind_at": "2026-03-21T20:00:00+08:00",
    "priority": null,
    "bucket": "next",
    "confidence": 0.94
  },
  "needs_confirmation": false,
  "questions": [],
  "dry_run": true
}
```

### 8.2 apply 模式：确认后写库

请求：

```json
{
  "input": "明晚8点提醒我给张三发合同",
  "context": {
    "timezone": "Asia/Shanghai",
    "source": "chat_ai",
    "source_ref": "om_xxx",
    "actor": "assistant"
  },
  "dry_run": false
}
```

响应建议：

```json
{
  "task": {
    "id": "uuid",
    "title": "给张三发合同",
    "bucket": "next",
    "status": "active",
    "due_at": "2026-03-21T20:00:00+08:00",
    "remind_at": "2026-03-21T20:00:00+08:00"
  },
  "draft": {
    "intent": "create_task",
    "summary": "给张三发合同",
    "bucket": "next",
    "confidence": 0.94
  },
  "needs_confirmation": false,
  "questions": [],
  "dry_run": false
}
```

### 8.3 错误响应建议

```json
{
  "error": {
    "code": "invalid_datetime",
    "message": "Invalid or ambiguous datetime expression",
    "details": [
      {
        "field": "remind_at",
```

## 10. 开发落地建议

## 10.1 建议拆成三层，不要把逻辑堆进 route

建议目录职责：

### A. parser

负责：

- 文本解析
- 时间短语抽取
- 标题抽取
- 产出 draft

### B. policy / planner

负责：

- 判断是否可直接创建
- 生成 bucket
- 判断是否需要追问
- 生成 questions

### C. sink / writer

负责：

- task/project 真正写库
- source/source_ref/actor 注入
- operation log

这样后面把启发式 parser 换成 LLM parser 时，不会改塌整条链路。

## 10.2 先固定规则，再提高智能度

v0.2.0 不要追求“全部靠模型自由发挥”。

建议顺序：

1. 固定 bucket 规则
2. 固定追问触发条件
3. 固定回复模板
4. 再提升时间解析覆盖率

## 10.3 当前实现的几个已知差距

对照当前代码，至少还有这些差距需要补：

1. 当前 `capture` 只有 `title/due_at/time_expression/confidence/bucket/status`，没有 `needs_confirmation/questions/intent`
2. 当前空输入会保底成 `Untitled task`，正式合同不应这样
3. 当前“明天/今天/下周”时间解析还比较粗，不足以支撑“提醒我”场景
4. 当前 `note=text.strip()` 是合理保底，但缺少“解析后的正式摘要”和“原文备注”的清晰区分
5. 当前尚未体现任务/项目分流能力

## 11. 一句话收敛结论

v0.2.0 的交互合同应收敛为：

> 明确任务直接建；模糊但可安全收集的进 Inbox；任务/项目/提醒时间存在关键歧义时必须追问；非法输入不建；所有结果都返回稳定的结构化 draft、追问列表或错误码。

这份合同一旦定住，后续无论 parser 是规则、LLM 还是混合实现，前后端与测试都可以围绕同一套行为标准推进。
